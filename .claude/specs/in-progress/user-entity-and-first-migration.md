# Spec: user-entity-and-first-migration
**Status:** in-progress
**Owner:** @arul.goyal
**Started:** 2026-06-14

## Goal
Design the `User` table for Cashroom (Step 3): a `User` TypeORM entity plus the
project's **first** migration. This also means standing up the migration
*infrastructure* that does not yet exist (a standalone `DataSource` for the CLI
and npm scripts). **No auth logic** тАФ entity + migration only. The spec doubles
as the teaching artifact: each field, the role enum, migrations vs `synchronize`,
and the team-conflict scenario are explained inline.

---

## Decisions (locked with user)
- [x] **PK type:** `bigint` auto-increment (not UUID).
- [x] **Role storage:** single `varchar` column + TypeScript `enum` + DB `CHECK`
      constraint. *Not* a join table, *not* a `text[]` array, *not* `jsonb`.
      (Rationale below тАФ roles graduate to a lookup table only when a role needs
      to own permissions/metadata.)
- [x] **Shared base:** create an abstract `BaseEntity` now (`id`, `createdAt`,
      `updatedAt`); `User` extends it. Future `Loan` etc. reuse it.
- [x] **No auth logic** this step.

## Part 1 тАФ The `User` entity, field by field

Entity maps to table **`users`**. `bigint` columns are returned by TypeORM as
**`string`** in JS (a deliberate guard against precision loss beyond 2^53) тАФ so
`id` is typed `string`, not `number`. This is consistent with the project rule
that money is `bigint`; same reasoning, same JS-string handling.

### `id` тАФ `BaseEntity`
- **Why it exists:** every row needs a stable, unique handle for FKs and lookups.
- **Why named `id`:** universal convention; TypeORM/RBAC tooling expects it.
- **Decorator:** `@PrimaryGeneratedColumn('increment', { type: 'bigint' })` тАФ
  DB-generated sequential key. Chose `increment` over `uuid` per decision;
  `bigint` (not `int`) gives headroom and matches the codebase's money typing.
- **SQL:** `id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
  (TypeORM 1.x emits an identity/serial-backed bigint PK).

### `email`
- **Why it exists:** the login identifier and primary contact.
- **Why named `email`:** plain and unambiguous.
- **Decorator:** `@Column({ type: 'varchar', length: 255, unique: true })`.
  `unique: true` creates a unique index тАФ prevents duplicate accounts *and*
  speeds up the by-email lookup that every login will do. Store **lowercased**
  at the service layer later so `A@x.com` and `a@x.com` can't both register.
- **SQL:** `email varchar(255) NOT NULL`, plus `UNIQUE INDEX (email)`.

### `passwordHash`
- **Why it exists:** authentication needs a verifier for the password.
- **Why named `passwordHash`, not `password`:** we **never** store the plaintext
  password. The name encodes the invariant тАФ what lives here is a one-way
  bcrypt/argon2 hash. A column called `password` invites someone to write the
  raw value; `passwordHash` makes the wrong thing read as obviously wrong.
- **Decorator:** `@Column({ type: 'varchar', length: 255, select: false })`.
  `select: false` keeps the hash **out of default queries**, so it won't leak
  into API responses unless explicitly requested. (Hashing itself is Step тЙе4.)
- **SQL:** `password_hash varchar(255) NOT NULL`.

### `role`
- **Why it exists:** authorization тАФ decides what the user may do.
- **Why named `role` (singular):** one role per user (decision); plural would
  imply an array we deliberately chose not to build.
- **Decorator:** `@Column({ type: 'varchar', length: 32, default: UserRole.STUDENT })`
  typed as the `UserRole` enum. Stored as a string; a DB `CHECK` constraint
  (added in the migration) enforces the allowed set at the database level too.
- **SQL:** `role varchar(32) NOT NULL DEFAULT 'student'`
  `CONSTRAINT chk_users_role CHECK (role IN ('student','admin'))`.

### `isEmailVerified`
- **Why it exists:** gates actions until the user proves they own the address.
- **Why named `isEmailVerified`:** `isтАж` prefix reads as a boolean at call sites.
- **Decorator:** `@Column({ type: 'boolean', default: false })`.
- **SQL:** `is_email_verified boolean NOT NULL DEFAULT false`.

### `createdAt` / `updatedAt` тАФ `BaseEntity`
- **Why they exist:** audit/debugging and ordering ("newest users", "stale rows").
- **Why named this way:** camelCase entity props; snake_case columns.
- **Decorators:** `@CreateDateColumn()` (set once on insert) and
  `@UpdateDateColumn()` (auto-bumped by TypeORM on every save). Using the
  dedicated decorators rather than plain columns means we don't hand-manage
  timestamps.
- **SQL:** `created_at timestamptz NOT NULL DEFAULT now()`,
  `updated_at timestamptz NOT NULL DEFAULT now()`.

> Column naming: entity props are camelCase, DB columns snake_case. We'll set a
> `SnakeNamingStrategy` **or** annotate `name:` per column. Decision deferred to
> implementation тАФ flagged in Open Questions.

## Part 2 тАФ The role enum & RBAC

```ts
export enum UserRole {
  STUDENT = 'student',
  ADMIN   = 'admin',
}
```

- **Why store the role in the DB at all (vs hardcoding in code)?**
  The *list of allowed role types* lives in code (the enum) тАФ that's fine and
  cheap to extend. But **which role a given user has** is per-user *data*, so it
  must live in the DB. You can't hardcode "Arul is an admin" in source; it
  changes per user and at runtime (promoting/demoting a user is an `UPDATE`, not
  a code deploy). So: the *vocabulary* is in code, the *assignment* is in the row.
- **How this connects to RBAC:** Role-Based Access Control = permissions attach
  to **roles**, and users get permissions by **holding a role**. This `role`
  column is the link between a user and their permission set. Later (auth step),
  a NestJS guard reads `user.role` (typically from a JWT claim, so no DB hit per
  request) and allows/denies the route. Today we only lay the data foundation.
- **Why varchar+enum now, not a roles table:** for `student | admin` with no
  per-role metadata, a lookup table adds a table + join to store what is just a
  string. Promote to a `roles` table the day a role must carry permissions тАФ
  that's the right RBAC moment, not now. A `CHECK` constraint gives us DB-level
  safety in the meantime, and adding a new role *type* later is a one-line enum
  edit plus a small `CHECK`-altering migration.

## Part 3 тАФ Migrations

- **What is a migration?** A versioned, ordered script (`up()` to apply, `down()`
  to undo) describing an explicit schema change. Migrations are committed to git,
  run in sequence, and TypeORM records which have run in a `migrations` table тАФ
  so every environment (your laptop, CI, prod) converges to the same schema.
- **Why migrations instead of `synchronize: true`?** `synchronize: true` diffs
  entities against the live DB on every boot and auto-applies changes тАФ which can
  **silently DROP columns/tables and lose data**. Unacceptable for a lending
  ledger. The codebase already sets `synchronize: false`
  (`src/database/database.module.ts:37`, with that exact rationale in a comment).
  Migrations make every schema change **reviewable, ordered, and reversible**.

### Infrastructure to build (none exists yet)
1. **`src/database/data-source.ts`** тАФ a standalone `DataSource` the TypeORM CLI
   can import. Nest's `forRootAsync` config is invisible to the CLI (no DI at the
   command line), so the CLI needs its own `DataSource` that reads the same
   `DATABASE_*` env vars and points at the entity + migration globs.
   Loads env via `dotenv` (available transitively through `@nestjs/config`; will
   add it explicitly as a devDependency to be safe).
2. **npm scripts** in `package.json` (using the installed `typeorm-ts-node-commonjs`
   binary so migrations run as TS without a build step). Exact flags will be
   confirmed against `npx typeorm --help` at implementation time (TypeORM is at
   1.0.0 here, not the more-documented 0.3.x):
   - `migration:generate` тЖТ diff entities vs DB, write a timestamped migration
   - `migration:run` тЖТ apply pending migrations
   - `migration:revert` тЖТ roll back the last applied migration
   - `migration:show` тЖТ list applied/pending
3. **Generate** the first migration тЖТ `тАж-CreateUsersTable.ts` (creates `users`,
   the unique email index, and the `chk_users_role` CHECK constraint). We will
   **read the generated SQL** before running it, and hand-add the `CHECK`
   constraint if the generator doesn't emit it.
4. **Run & verify** the table exists.
5. **Show rollback.**

## Part 4 тАФ Team scenario: two devs edit the same entity, no migrations

With `synchronize: true` and no migrations:
- Dev A adds `phoneNumber`, Dev B renames `isEmailVerified` тЖТ `emailVerified`.
  Each boots locally and `synchronize` reshapes *their* DB to match *their*
  branch. **No shared, ordered record of what changed.**
- On merge, there's no migration to reconcile. Whoever boots next has
  `synchronize` auto-alter the DB to match merged entities тАФ possibly **dropping
  a column** (and its data) that the other relied on. No review, no rollback.
- Staging/prod drift: each environment's schema depends on *which entity version
  booted last*, not on an agreed sequence. Bugs become "works on my machine."

With migrations: each dev commits a timestamped migration. Git surfaces the
ordering, the diff is **reviewed in the PR**, conflicts are explicit, runs are
recorded in the `migrations` table, and any change is reversible via `down()`.

## Validation
- **Build/lint:** `npm run lint` and a `nest build` succeed (entity compiles, decorators resolve).
- **Migration generates:** `migration:generate` produces a `CreateUsersTable` migration; we read its `up()` SQL and confirm it creates `users`, the unique `email` index, and `chk_users_role`.
- **Migration runs:** `migration:run` applies cleanly; `migration:show` lists it as `[X]`.
- **Table exists in Postgres:** query the live DB (e.g. via `docker exec <postgres-service> psql -U cashroom -d cashroom_dev -c "\d users"`) and confirm columns, types, the unique index, and the CHECK constraint. *(Exact container/service name to confirm against root docker-compose at implementation.)*
- **Rollback works:** `migration:revert` drops `users`; `migration:show` shows it pending again.
- **App still boots:** `npm run start:dev` connects with `autoLoadEntities` discovering `User` (no `synchronize` side effects).

## Rollback
- Schema: `npm run migration:revert` (runs the migration's `down()` тЖТ drops `users`).
- Code: delete the new files and the `forFeature([User])` line; revert `package.json`. No data at risk (fresh table, no auth/data flows yet).

## Risk: LOW
New tables only, no existing data, `synchronize` stays off, no auth/business logic touched. Main new surface is the migration tooling, which is fully reversible and verified before/after running.
